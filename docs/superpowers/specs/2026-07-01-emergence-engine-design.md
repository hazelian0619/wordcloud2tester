# 设计规格：WordCloud Emergence —— 创意人的灵感器

**Feature Branch**: `spec/emergence-engine`
**Created**: 2026-07-01
**Status**: Draft — 待用户 review
**方法论**: 借用 spec-kit 的 Constitution + Acceptance Scenario 结构，遵循 superpowers brainstorming 的成功标准优先原则

---

## 0. 一句话定位

> **这个产品不给你答案，它给你「你自己想不到、但一看就知道能用」的中间物，让创意人在两个概念的交叉处撞出火花。**

目标用户：**创意人**（广告 / 设计 / 写作 / campaign 策划）。
不是通用概念工具，不是教学启发器。所有设计向「给创意人一个惊艳、可落地的创意锚点」对齐。

---

## 1. Constitution（项目宪法 · 不可违背的核心原则）

### 原则一：涌现优先（NON-NEGOTIABLE）
每一次扩展都必须产生用户**自己想不到**的东西。若结果全是近义词 / 教科书目录（如「时空 → 时间维度、空间维度」），即判为**失败**，无论技术上多正确。

### 原则二：为创意人服务，不为「正确」服务
权重不代表「相似度」，代表「**惊喜度 × 可用性**」。越像核心词越无聊，分越低；越意外又越能用，分越高。

### 原则三：交叉处才有价值
产品的灵魂动作是「取两个概念，求它们之间的第三者」。单点发散是辅助，交集涌现是主线。

### 原则四：一切可验收，不靠感觉
每个功能上线前必须能用下面的 Acceptance Scenario 客观判定成功 / 失败。

### 原则五：小步验证，先地基后装修
先证明引擎能出金子（第1层），再做交互和体验（第2、3层）。禁止在塌陷的地基上装修。

## 2. 现状诊断（Review 结论 · 洋葱模型由内而外）

| 层 | 现状 | 与理想的差距 | 优先级 |
|----|------|-------------|--------|
| 第1层 引擎 | 提示词求「语义相关词」，只吐最近邻居 | 应求「意外的第三者」，出金子 | **P1 病根** |

## 3. User Stories（按优先级 · 每个都是独立可交付的 MVP 切片）

### User Story 1 —— 单词涌现引擎（Priority: P1）

**用户旅程**：创意人输入一个模糊方向（如「时空」），期待拿到 8 个**自己想不到、但一看就能用**的意外锚点，而不是教科书同义词。

**Why this priority**：这是病根，也是唯一不可替代的价值。引擎不出金子，其余全白搭。改动最小、见效最猛。

**Independent Test**：仅改后端提示词即可独立测试，无需动前端。输入若干测试词，用下方 Acceptance Scenario 判定。

**Acceptance Scenarios**：
1. **Given** 输入「时空」，**When** 调用扩展，**Then** 8 个结果里**至少 3 个**是跨领域 / 意外的（如「乡愁、老照片、初恋」），**不能**全是物理教科书词（「时间维度、空间维度」）。
2. **Given** 任一核心词，**When** 看权重排序，**Then** 权重最高的**不是最像的同义词**，而是「最惊艳又最能用」的那个（惊喜度×可用性）。
3. **Given** 8 个结果，**When** 检查语义维度，**Then** 它们**散布在多个维度**（人物/意象/对立面/通感…），不全挤在一个角落。
4. **Given** 每个结果词，**When** 创意人读它，**Then** 能勾起一个**具体形象或画面**，而非纯抽象概念。

### User Story 2 —— 交集涌现（Priority: P1，紧随 Story 1）

**用户旅程**：创意人在画布上选中两个词（如「龙」+「少女」），点「求涌现」，工具给出**同时活在两个词里的意外第三者**（美杜莎、小龙女、哪吒…）。这是 README 的灵魂场景。

**Why this priority**：产品叫「涌现」，却没有涌现的操作入口。这是创意人最想要的动作。

**Independent Test**：前端已有「连线模式」选点基础可复用；后端加一个「双概念交集」提示分支即可独立测试。

**Acceptance Scenarios**：
1. **Given** 选中「龙」+「少女」，**When** 求涌现，**Then** 结果**同时咬住两个词**（既有龙性又有少女性），而不是只扩展其中一个。
2. **Given** 任意两词，**When** 求涌现，**Then** 至少半数结果是「大多数人第一反应想不到」的意外形象。

### User Story 3 —— 灵感卡片收藏（Priority: P2）

**用户旅程**：创意人看到惊艳结果，一键「钉住」成灵感卡片，可加注解、可导出去用。

**Why this priority**：创意人要「抓住火花」，不要「回看路径」。故路径可视化降级，收藏升为主。

**Independent Test**：纯前端 + localStorage，独立于引擎可测。

**Acceptance Scenarios**：
1. **Given** 任一结果节点，**When** 点「钉住」，**Then** 生成一张含概念+来源+可编辑注解的卡片并持久化（刷新不丢）。
2. **Given** 已收藏若干卡片，**When** 导出，**Then** 得到可直接用于 campaign 的文本 / Markdown。

## 4. 技术设计（对齐现有代码，最小侵入）

### 4.1 引擎改造（Story 1）—— `api/path-expand.py`
- **人格重写**：system prompt 从「语义概念扩展专家」→「资深创意总监，专门寻找意外的跨界形象」。
- **任务重定义**：显式要求「同时活在核心概念里、且大多数人第一反应想不到」，明确**拒绝近义词**、要求跨文化跨领域。
- **权重语义翻转**：`weight = 惊喜度 × 可用性`（模型输出），不再是相似度。字号最大 = 最惊艳可用。
- **多维强制**：提示词要求结果散布在不同维度，避免同质化。
- **兼容性**：保留已修好的 `max_completion_tokens` 降级逻辑，不回退。

### 4.2 交集涌现（Story 2）—— 前后端
- **后端**：`path-expand.py` 增加分支——当 `current_concept` 含多词（或新增 `concept_pair` 字段）时，走「双概念交集」提示词。
- **前端**：复用现有「连线模式」的两点选择交互，改造出「求涌现」动作，命中后调用交集分支并把第三者作为新节点落在两词中间。

### 4.3 灵感卡片（Story 3）—— `frontend/index.html`
- 节点上加「钉住」入口 → 存 `localStorage` key `wc_discoveries`。
- 「我的灵感」抽屉列出卡片，支持编辑注解 / 删除 / 导出。
- localStorage 不可用时降级为内存态。
- i18n：新文案中英各一套。

### 4.4 顺带修复（工作中发现，与目标相关）
- dev server 冷启动竞态：前端点击后 5-6 秒无反馈 → 需加载态提示（并入 Story 期间处理）。

---

## 5. 验证方式

每完成一个 Story，我实际操作并对照 Acceptance Scenario 逐条报告成功/失败：
- Story 1：跑「时空 / 孤独 / 潮汕菜」改前改后 A/B，你亲眼数惊喜度。
- Story 2：跑「龙+少女」「咖啡+雨」，看是否咬住两词。
- Story 3：钉一张卡片 → 刷新 → 确认还在 → 导出。

---

## 6. 明确不做（YAGNI）
- 不做真正的 GAT / PyTorch 神经网络（README 里的说法是营销，实际是提示词工程，本 spec 不假装有它）。
- 不做复杂的路径回放 / 多路径对比可视化（创意人不需要）。
- 不安装 spec-kit 脚手架进本项目（只借方法论，不引负担）。
- 不做用户账号 / 云端同步（本地 localStorage 足够验证价值）。
